# CAW Contract

## Current Adapter Boundary

`lib/caw/gateway.ts` defines:

- `connectWallet`
- `createPact`
- `executeCreditsPurchase`

Business services must use these methods instead of importing a CAW SDK directly.

## Real CAW Information Needed

Before production wiring, collect:

- API base URL and environment.
- API key/secret format and signing requirements.
- Wallet connect or pairing endpoint.
- Pact creation endpoint and exact policy schema.
- Pact lifecycle states and polling/webhook behavior.
- Contract call execution endpoint.
- Transaction status endpoint or webhook.
- Policy denial error format.

## Expected Contract Call

The CAW-executed transaction should call:

```solidity
buyCredits(bytes32 orderId, address creditAccount, uint256 amountUsdc)
```

Expected constraints:

- `orderId` is a backend-generated bytes32 id derived from the top-up order.
- `creditAccount` should map to the user or wallet receiving credits.
- `amountUsdc` uses USDC minor units with 6 decimals.
- The target contract address must match the Pact allowlist.
- The USDC token address must match the configured network.

## Error Handling

Normalize CAW failures into stable application reasons:

- `missing_caw_authorization`
- `authorization_pending_user_approval`
- `authorization_expired`
- `policy_denied`
- `wallet_insufficient_funds`
- `network_unavailable`
- `transaction_reverted`
- `transaction_status_unknown`

Unknown CAW errors should leave orders retry-safe and should not credit balances.
